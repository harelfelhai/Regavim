# תוכנית אבטחת מידע — מערכת Regavim

**גרסה:** 1.0 | **תאריך:** יוני 2026 | **סטטוס:** טיוטה לאישור הנהלה

מסמך זה מגדיר תוכנית אבטחה עקרונית למערכת Regavim, ביעד עמידה בסטנדרט הנדרש
ממערכות מידע רפואיות בישראל — הרף הרגולטורי הגבוה ביותר הנהוג למאגרי מידע רגיש.
התוכנית מפרטת את הבקרות והכלים בכל שלב של זרימת הנתונים, באחסון, בזיהוי משתמשים,
בתשתית ובתהליכי הפיתוח, ומסתיימת בטבלת פערים (מצב קיים מול יעד) ומפת דרכים ליישום.

## 1. מסגרת רגולטורית ותקנים

המערכת תיושר מול המסגרות הבאות:

| מסגרת | רלוונטיות |
|---|---|
| **תקנות הגנת הפרטיות (אבטחת מידע), התשע"ז-2017** | מסגרת החובה המרכזית. מערכת רפואית מסווגת כמאגר ב**רמת אבטחה גבוהה** — זהו הרף שאליו מיישרים. |
| **חוק הגנת הפרטיות + תיקון 13 (בתוקף מאוגוסט 2025)** | חובות דיווח, מינוי ממונה הגנת פרטיות (DPO) בתנאים מסוימים, סמכויות אכיפה מוגברות של הרשות להגנת הפרטיות. |
| **חוזרי מנכ"ל משרד הבריאות בנושא הגנת סייבר במוסדות רפואיים** | דרישות ייעודיות: ניהול סיכונים, הפרדת רשתות, ניטור, תוכנית התאוששות. |
| **ISO/IEC 27001 + ISO 27799** | מערכת ניהול אבטחת מידע (ISMS) והחלתה על מידע בריאות — בסיס מתודולוגי לבקרות. |
| **תורת ההגנה של מערך הסייבר הלאומי** | מתודולוגיה ארגונית להערכת סיכונים ובניית הגנה בשכבות. |
| **OWASP ASVS / OWASP Top 10** | רף טכני לאבטחת האפליקציה עצמה. |

**משמעות "רמת אבטחה גבוהה" בתקנות (עיקרי הדרישות):** מסמך הגדרות מאגר; נוהל
אבטחה מחייב; סקר סיכונים ומבדקי חדירות **אחת ל-18 חודשים**; ביקורת אבטחה
חיצונית **אחת ל-24 חודשים**; ניהול הרשאות מתועד; תיעוד אירועי אבטחה ושמירת
לוגים; דיווח מיידי לרשות להגנת הפרטיות על אירוע אבטחה חמור; הצפנת תקשורת
ברשת ציבורית; בקרות על מיקור-חוץ (ספקי ענן).

## 2. סיווג המידע ומיפוי המאגר (תקנה 2 — מסמך הגדרות מאגר)

| סוג מידע | רגישות | היכן נשמר |
|---|---|---|
| פרטי משתמשים (אימייל, hash סיסמה, תפקיד) | גבוהה | טבלת `users` (PostgreSQL/Neon) |
| מיקומי GPS של מדווחים (`user_lat/lng`) | **גבוהה — מידע מיקום של אדם מזוהה** | טבלת `reports` |
| תמונות ראיה + EXIF (כולל GPS, זמן, מכשיר) | **גבוהה — ראיה משפטית, מיקום, מטא-דאטה אישי** | Cloudinary (prod) / קבצים מקומיים (dev) |
| תוכן דיווחים, קטגוריות, סטטוסים | בינונית | `reports` |
| היסטוריית תלונות לרשויות | בינונית-גבוהה | `complaint_submissions` |
| סודות תפעול (JWT key, מפתחות API, SMTP) | קריטית | משתני סביבה בלבד |

**עקרונות מחייבים:** מזעור מידע (לא אוספים מה שלא נדרש), הגדרת תקופות שמירה
(retention) לכל סוג, ורישום מי רשאי לגשת למה. יש לתחזק מסמך הגדרות מאגר עדכני
הכולל את הטבלה הזו, מטרות העיבוד, והעברות לצדדים שלישיים (Anthropic, Cloudinary, ספק SMTP).

## 3. עקרונות-על ארכיטקטוניים

1. **הגנה בשכבות (Defense in Depth)** — אין הסתמכות על בקרה יחידה; כשל בשכבה אחת נבלם בשכבה הבאה.
2. **הרשאה מינימלית (Least Privilege)** — כל משתמש, שירות ומפתח API מקבל את המינימום הנדרש.
3. **הצפנה מקצה לקצה של נתונים במנוחה ובתנועה** — TLS 1.2+ בכל ערוץ; הצפנה at-rest בכל מאגר.
4. **Human-in-the-loop נשמר גם באבטחה** — פעולות רגישות (מחיקה, הגשת תלונה) דורשות תפקיד מתאים ומתועדות ב-audit log.
5. **Fail Secure** — כשל בבקרה (למשל אימות טוקן) חוסם גישה, לעולם לא מאפשר אותה.
6. **שקיפות ותיעוד** — כל גישה למידע רגיש ניתנת לשחזור: מי, מתי, מה.

## 5. זיהוי, אימות וניהול הרשאות (תקנות 7–8)

| בקרה | יעד | מצב קיים |
|---|---|---|
| Hash סיסמאות | bcrypt (cost ≥ 12) | ✅ קיים |
| מדיניות סיסמאות | אורך ≥ 12, בדיקה מול רשימות סיסמאות פרוצות (HIBP API), ללא דרישות הרכב שרירותיות | ❌ חסר |
| אימות דו-שלבי (MFA) | **חובה ברף רפואי** למנהלים ואדמינים לפחות — TOTP (pyotp) או WebAuthn | ❌ חסר |
| תוקף טוקן | **קיצור מ-30 יום ל-15–60 דקות** + Refresh Token עם רוטציה ו-revocation ב-DB | ❌ פער מהותי |
| נעילת חשבון | נעילה זמנית אחרי 5 כשלונות התחברות + השהיה מדורגת | ❌ חסר |
| RBAC | שלוש רמות (coordinator/manager/admin) + **בדיקות ownership פר-משאב** | ⚠️ חלקי — אין ownership |
| ניהול הרשאות מתועד | רישום מתן/שלילת הרשאה (מי אישר, מתי) — דרישת תקנה 7 | ❌ חסר |
| SECRET_KEY | ולידציה בעליית השרת: סירוב לעלות ב-production עם מפתח placeholder/קצר מ-32 בתים; רוטציה מתועדת | ❌ ברירת מחדל חלשה |
| הסרת משתמשים | נוהל offboarding: השבתת חשבון = ביטול כל הטוקנים הפעילים | ❌ חסר (מחייב revocation) |

## 6. אבטחת בסיס הנתונים והאחסון

### PostgreSQL (Neon — production)
- **תעבורה:** `sslmode=require` (עדיף `verify-full` עם CA) ב-`DATABASE_URL` — חובה, כיום לא נאכף.
- **At rest:** Neon מצפין דיסקים כברירת מחדל (AES-256) — לתעד כבקרת ספק; בנוסף, **הצפנה אפליקטיבית ברמת עמודה** לשדות הרגישים ביותר (קואורדינטות GPS של המדווח) באמצעות `cryptography` (Fernet/AES-GCM) עם מפתח נפרד מה-SECRET_KEY.
- **הרשאות DB:** משתמש אפליקציה ללא DDL (מיגרציות בלבד עם משתמש נפרד); ללא superuser.
- **גיבויים:** גיבוי יומי אוטומטי, מוצפן, עם בדיקת שחזור רבעונית מתועדת (דרישת חוזרי משרד הבריאות — DR). PITR של Neon + ייצוא מוצפן חיצוני.
- **SQL Injection:** מוגן ע"י SQLAlchemy parameterized queries — ✅ קיים; להימנע לעד מ-raw SQL עם שרשור מחרוזות.

### SQLite (dev)
- לפיתוח בלבד; **אסור** שיכיל נתוני אמת. קובץ ה-DB ב-`.gitignore`.

### Cloudinary (תמונות)
- מעבר ל-**authenticated/private delivery type** + **signed URLs** עם תפוגה קצרה במקום URL-ים ציבוריים.
- הגבלת ה-API key להרשאות upload/admin הנדרשות בלבד; רוטציית מפתחות מתועדת.
- תיעוד Cloudinary כ"מחזיק מידע" (מיקור חוץ — תקנה 15) כולל DPA.

### Retention ומחיקה
- מדיניות שמירה מוגדרת: דיווחים שנדחו — X שנים; לוגים — שנה לפחות (דרישת תיעוד); תמונות יתומות (ללא דיווח) — מחיקה אוטומטית אחרי 30 יום.
- הליך מימוש "זכות למחיקה/עיון" לפי חוק הגנת הפרטיות.

## 7. אבטחת האפליקציה וה-API

| בקרה | יישום | עדיפות |
|---|---|---|
| **Rate limiting** | `slowapi`: login — 5/דקה/IP; register — 3/שעה/IP; upload — 20/שעה/משתמש; complaints — 10/יום/משתמש | 🔴 קריטי |
| **Security headers** | middleware המוסיף: `Strict-Transport-Security`, `X-Content-Type-Options: nosniff`, `X-Frame-Options: DENY`, `Content-Security-Policy`, `Referrer-Policy: strict-origin-when-cross-origin` | 🔴 קריטי |
| **CORS** | רשימת origins מפורשת; **ולידציה בעליית השרת שדוחה `*` כשסביבה=production** | 🔴 קריטי |
| ולידציית קלט | Pydantic + `extra="forbid"` + GPS bounds — ✅ קיים; להוסיף מגבלות אורך לשדות טקסט חופשי (description, tags) | 🟡 |
| הודעות שגיאה | ללא stack traces או פרטי DB בתשובות ללקוח (exception handler גלובלי) | 🟡 |
| תלות ב-UUID כסוד | לבטל — כל משאב מאחורי אימות והרשאה (ראו 4ה') | 🔴 |
| TrustedHost | `TrustedHostMiddleware` עם domains מורשים בלבד | 🟡 |

## 8. תשתית, ענן ומיקור חוץ (תקנה 15)

| ספק | תפקיד | בקרות נדרשות |
|---|---|---|
| Render (backend) | compute | הזרקת סודות דרך env בלבד; הגבלת גישה לחשבון עם MFA; לוגים ללא סודות |
| Vercel (frontend) | static hosting | HTTPS + HSTS אוטומטי; הגבלת חברי צוות |
| Neon (DB) | PostgreSQL | TLS, IP allowlist אם זמין, גיבויים |
| Cloudinary | תמונות | signed delivery, רוטציית מפתחות |
| Anthropic | ניתוח AI | DPA, אי-שימוש בנתונים לאימון |
| ספק SMTP/Resend | דוא"ל | TLS, App Password ייעודי |

- **רישום ספקים:** טבלה זו מתועדת במסמך הגדרות המאגר ונסקרת שנתית.
- **גישת מפתחים ל-production:** מינימלית, אישית (לא חשבון משותף), עם MFA, ומתועדת.
- **הפרדת סביבות:** dev/staging/prod עם סודות נפרדים לחלוטין; אין נתוני אמת מחוץ ל-prod.

## 9. ניטור, לוגים ו-Audit Trail (תקנה 10)

- **Audit log אפליקטיבי** (טבלה ייעודית או append-only log): התחברות (הצלחה/כשל), צפייה בדיווח/תמונה, שינוי סטטוס, מחיקה, הגשת תלונה, שינוי הרשאות. שדות: מי, מה, מתי, מאיפה (IP), על איזה משאב.
- **שמירת לוגים:** שנתיים לפחות (רף רמה גבוהה), במאגר ש-write-only עבור האפליקציה.
- **ניטור חריגות:** התראות על ריבוי כשלונות התחברות, גישה בשעות חריגות, הורדות מסיביות של תמונות. כלים: Sentry (שגיאות) + לוגי Render + התראות בסיסיות; בהמשך — שירות SIEM קל (למשל Grafana Cloud / Better Stack).
- **היגיינה:** אסור שיופיעו בלוגים — סיסמאות, טוקנים, מפתחות, תוכן EXIF מלא.

## 10. ניהול אירועי אבטחה והמשכיות (תקנה 11 + DR)

1. **נוהל תגובה לאירוע (IR):** זיהוי → בלימה (ביטול טוקנים, רוטציית מפתחות, ניתוק ספק) → חקירה (לוגים) → התאוששות → הפקת לקחים. בעל תפקיד מוגדר (ממונה אבטחה).
2. **חובת דיווח:** אירוע אבטחה חמור במאגר ברמה גבוהה מחייב **דיווח מיידי לרשות להגנת הפרטיות**; תיקון 13 מרחיב חובות יידוע לנושאי המידע.
3. **DR/גיבויים:** RPO ≤ 24 שעות (גיבוי יומי), RTO מוגדר ונבדק; תרגול שחזור רבעוני מתועד.
4. **רשימת אנשי קשר** ותבנית דיווח מוכנות מראש.

## 13. מפת דרכים ליישום

**שלב 1 — "סגירת דלתות" (שבועות 1–2), פערים P0:**
rate limiting, אימות על קבצים, ולידציית SECRET_KEY ו-CORS בעליית שרת,
security headers, קיצור תוקף טוקן + refresh, בדיקות ownership.
כולו קוד backend ממוקד — ללא תלות בספקים.

**שלב 2 — הקשחה (שבועות 3–6), פערים P1:**
TLS ל-DB, MFA, audit log + ניטור (Sentry), נעילת חשבון ומדיניות סיסמאות,
שילוב כלי האבטחה ב-CI, signed URLs ב-Cloudinary.

**שלב 3 — עמידה רגולטורית מלאה (חודשים 2–4), פערים P2:**
הצפנת עמודות רגישות, מסמך הגדרות מאגר + נוהל אבטחה + נוהל IR כתובים,
הסכמי DPA עם ספקים, מדיניות retention ממומשת בקוד, סריקת נוזקות.

**שלב 4 — מחזורי (קבוע):**
סקר סיכונים + מבדק חדירות כל 18 חודשים; ביקורת חיצונית כל 24 חודשים;
תרגול שחזור רבעוני; סקירת הרשאות וספקים שנתית; רוטציית מפתחות חצי-שנתית.
